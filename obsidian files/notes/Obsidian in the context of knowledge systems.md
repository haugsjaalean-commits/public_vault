# Mapping my theory onto Obsidian 


## How my theory maps onto my current method


My current method is efficient and fast for some things, but it falls apart logically when I start to compare it to my theory. This isn't necessarily a bad thing as long as I can keep things efficient and flowing and connected. 

---

### Search pathways 
The greatest logical hole (but also the greatest strength) of my current method is the fact that each note acts both as a KU and a GP. *O* is the first GP. All notes leading out of O are KU, as they represent things or concepts that exist in the real world, but they are also GPs, as they link to the subsequent set of notes. This logically brakes away from my theory, which requires a distinct separation between the GPs and the KUs. Is it the end of the world that there is a logical discrepancy like this? I don't know. I think that it can be fine, but I also feel that it causes problems later down the road. 

### [[bases|Bases]] and [[metadata]]


I described the act of using metadata in order to effectively create classes. When we do this, we are basically using [[tags]] to create theoretical KUs. These KUs are arranged in a structure of inheritance. Then, using bases, we are able to create queries which mimic quite well the *edges* in my theory. This is all well and good, but the only problem is that these hierarchies which we are creating with the tags are completely rigid like a folder structure. This is bad, because we don't want to ever fall into this trap of rigidity; we always want to be flexible.
